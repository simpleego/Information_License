# 형상관리 도구
- csv(Concurrent Versions System) 
- svn(Subversion)
- Git(Global Information Tracker)

---

## 📊 CVS, SVN, Git 비교

| 구분 | CVS (Concurrent Versions System) | SVN (Subversion) | Git |
|------|----------------------------------|------------------|-----|
| **방식** | 중앙집중형(CVCS) | 중앙집중형(CVCS) | 분산형(DVCS) |
| **장점** | - 초기 버전 관리 도구로 단순<br>- 기본적인 협업 가능 | - CVS의 단점을 개선<br>- 대규모 바이너리 파일 관리에 유리<br>- 초보자 친화적 | - 분산형 구조로 오프라인 작업 가능<br>- 브랜치/병합이 빠르고 유연<br>- 속도와 성능 우수 |
| **단점** | - 브랜치/병합 기능 미약<br>- 대규모 프로젝트에 부적합<br>- 유지보수 중단 | - 중앙 서버 의존 → 서버 다운 시 전체 작업 중단<br>- 브랜치/병합이 Git보다 느리고 복잡 | - 개념이 복잡해 학습 곡선이 높음<br>- 초보자에게 진입 장벽 |
| **난이도** | 쉬움 (기능 단순) | 중간 (CVS보다 개선, 직관적) | 어려움 (분산형 개념과 다양한 기능 학습 필요) |
| **사용 사례** | 과거 소규모 프로젝트 | 기업 내부 레거시 시스템, 문서 관리 | 현대 소프트웨어 개발 표준, 오픈소스 프로젝트 |
| **개발 시기** | 1990년 | 2000년 | 2005년 |

---

## 🔎 분석
- **CVS**: 가장 오래된 도구로 현재는 거의 사용되지 않음. 단순하지만 기능 부족.  
- **SVN**: CVS의 개선판으로 중앙집중형 관리에 적합. 여전히 일부 기업에서 사용.  
- **Git**: 현재 사실상 표준. 분산형 구조 덕분에 **속도, 유연성, 협업**에서 압도적 우위.  

---

## ⚠️ 고려할 점
- **규모가 작은 프로젝트**나 단순 문서 관리 → SVN이 여전히 실용적일 수 있음.  
- **대규모 협업, 오픈소스, 현대 개발 환경** → Git이 사실상 필수.  
- CVS는 **역사적 의미**만 남아 있으며, 신규 프로젝트에는 적합하지 않음.  

---

**요약:**  
RCS(Revision Control System)는 가장 초기의 형상관리 도구 중 하나로, **파일 단위 버전 관리**를 지원합니다. CVS, SVN, Git 같은 후속 도구들과 비교하면 **단순하지만 협업 기능이 부족**하다는 차이가 있습니다.

---

## 🗂️ RCS란 무엇인가?
- **정식 명칭:** Revision Control System  
- **등장 시기:** 1980년대 초반, UNIX 환경에서 개발  
- **특징:**  
  - 개별 파일의 변경 이력을 관리  
  - 각 버전을 저장할 때 전체 파일을 저장하지 않고, **차이점(diff)**만 기록 → 저장 공간 절약  
  - 명령어 기반으로 동작 (CLI 중심)  
  - 주로 **개인 개발자**가 소규모 프로젝트에서 사용  

---

## 🔄 다른 도구들과의 차이점
| 도구 | 방식 | 특징 | 협업 지원 |
|------|------|------|-----------|
| **RCS** | 공유 폴더 기반, 파일 단위 | 변경 이력 관리, 차이점 저장 | 협업 기능 거의 없음 |
| **SCCS** | RCS와 유사 | UNIX 기반 초기 도구 | 제한적 |
| **CVS** | 클라이언트-서버 방식 | 중앙 저장소에서 관리, 여러 개발자 동시 작업 가능 | 기본 협업 지원 |
| **SVN(Subversion)** | 클라이언트-서버 방식 | CVS 개선판, 디렉토리 단위 관리, 브랜치/태그 지원 | 강력한 협업 |
| **Git** | 분산 저장소 방식 | 로컬 저장소 + 원격 저장소, 브랜치 관리 강력, 오프라인 작업 가능 | 현대 개발 표준, 대규모 협업 최적화 |

---

## 📌 핵심 차이
- **RCS는 파일 단위 관리**라서 프로젝트 전체를 관리하기 어렵습니다.  
- **CVS, SVN은 중앙 서버 기반**으로 여러 개발자가 동시에 작업 가능.  
- **Git은 분산형**이라 각자 로컬에서 자유롭게 작업 후 병합할 수 있어 현대 협업에 최적화.  

---

## ⚠️ RCS의 한계
- 프로젝트 규모가 커지면 관리가 어려움  
- 팀 협업 기능 부족 → 충돌 해결이 힘듦  
- 현재는 거의 사용되지 않고, **학습용 또는 역사적 의미**로만 언급됨  

---

👉 정리하면, **RCS는 소프트웨어 형상관리의 시초로서 파일 단위 버전 관리에 초점을 맞춘 도구**이고, **CVS·SVN·Git 같은 후속 도구들은 협업과 대규모 프로젝트 관리에 적합하게 발전**했다고 이해하면 됩니다.  

혹시 원하시면, 제가 **RCS 명령어 예시(`ci`, `co`, `rcsdiff`)**도 보여드릴까요?

👉 결론적으로, **CVS는 과거용, SVN은 중간 단계, Git은 현재와 미래의 표준**이라고 이해하면 됩니다.
