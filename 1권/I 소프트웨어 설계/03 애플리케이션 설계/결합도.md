# 결합도
> **결합도(Coupling)**는 모듈과 모듈 사이의 상호 의존성을 나타내는 지표입니다.
> 응집도와 반대로, **결합도는 낮을수록(Low)** 모듈의 독립성이 높아져 유지보수와 재사용에 유리합니다.

---

## 1. 자료 결합도 (Data Coupling) - **가장 낮음**

모듈 간의 인터페이스가 오직 **데이터 요소(파라미터)**로만 구성된 경우입니다. 한 모듈이 다른 모듈을 호출할 때 필요한 값만 넘겨주고, 받은 모듈은 그 값을 처리해서 결과만 돌려줍니다.

* **예시:** `calculateSquare(int n)` 함수. 숫자 하나를 던져주면 제곱 값을 돌려받는 관계입니다.

## 2. 스탬프 결합도 (Stamp Coupling)

모듈 간에 **자료 구조(배열, 객체, 구조체 등)** 전체를 전달하는 경우입니다. 전달받은 모듈이 그 자료 구조의 일부 기능만 사용하더라도 전체를 넘겨야 하므로, 자료 구조가 바뀌면 영향을 받습니다.

* **예시:** '회원 가입' 함수에 '회원 객체' 전체를 넘기는 경우. (함수 안에서는 '이름'만 쓰더라도 객체 구조가 바뀌면 수정이 필요할 수 있습니다.)

## 3. 제어 결합도 (Control Coupling)

한 모듈이 다른 모듈의 상세한 **로직 제어 신호(Flag)**를 전달하여 "무엇을 할지" 결정해 버리는 상태입니다. 상위 모듈이 하위 모듈의 내부 구조를 너무 잘 알고 있어야 합니다.

* **예시:** `processOrder(isFastDelivery)` 함수. '빠른 배송 여부'라는 깃발(Flag)에 따라 내부의 if-else 문이 갈리는 경우입니다.

## 4. 외부 결합도 (External Coupling)

두 모듈이 외부에서 도입된 **데이터 포맷, 통신 프로토콜, 또는 디바이스 인터페이스**를 공유할 때 발생합니다. 외부 환경에 종속적이게 됩니다.

* **예시:** 특정 벤더의 데이터베이스 프로토콜이나 특정 외부 API의 형식을 여러 모듈이 공유하여 사용하는 경우.

## 5. 공통(공유) 결합도 (Common Coupling)

여러 모듈이 하나의 **전역 변수나 공통 데이터 영역**을 참조하고 갱신하는 상태입니다. 전역 변수 값이 어디서 바뀌었는지 추적하기 힘들어집니다.

* **예시:** 프로그램 전체에서 쓰이는 `public static int globalCounter`를 여러 클래스가 직접 수정하는 경우.

## 6. 내용 결합도 (Content Coupling) - **가장 높음**

한 모듈이 다른 모듈의 **내부 기능이나 데이터에 직접 접근**하여 수정하는 상태입니다. 한쪽을 고치면 반드시 다른 쪽도 고장 나게 되는 최악의 결합도입니다.

* **예시:** 클래스 A가 클래스 B의 `private` 멤버에 강제로 접근하거나, 내부 로직의 일부를 복사해서 사용하는 경우.

---

### 💡 결합도 요약 비교

| 등급 | 종류 | 상호작용 방식 | 평가 |
| --- | --- | --- | --- |
| **낮음** | **자료 (Data)** | 필요한 값만 전달 | **매우 좋음** |
| ↑ | **스탬프 (Stamp)** | 객체/자료 구조 공유 | **좋음** |
| ↑ | **제어 (Control)** | 제어 플래그 전달 | **보통** |
| ↓ | **외부 (External)** | 외부 데이터 형식 공유 | **주의** |
| ↓ | **공통 (Common)** | 전역 변수 공유 | **나쁨** |
| **높음** | **내용 (Content)** | 내부 로직/데이터 직접 참조 | **최악** |

> **설계 팁:** 결합도를 낮추려면 **"인터페이스를 최소화"**하고, 모듈이 서로의 내부 사정을 최대한 모르게(**정보 은닉**) 만들어야 합니다.
