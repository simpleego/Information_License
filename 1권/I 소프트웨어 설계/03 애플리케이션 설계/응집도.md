# 응집도
> **응집도(Cohesion)**는 하나의 모듈이 특정 목적을 위해 얼마나 밀집되어 있는지를 나타내는 척도입니다.
> 좋은 소프트웨어 설계를 위해서는 **"응집도는 높고(High), 결합도는 낮은(Low)"** 상태를 유지해야 합니다.

---

## 1. 우연적 응집도 (Coincidental Cohesion) - **가장 낮음**

모듈 내부의 요소들이 서로 아무런 관련 없이 묶여 있는 상태입니다.

* **예시:** `Utility` 클래스 하나에 '파일 출력', '로그인 체크', '수학 계산' 함수를 다 몰아넣은 경우.

## 2. 논리적 응집도 (Logical Cohesion)

유사한 성격의 기능을 수행하지만, 실제로는 서로 다른 일을 하는 요소들이 논리적으로 묶인 상태입니다. 실행 시점에 전달되는 파라미터에 따라 동작이 결정됩니다.

* **예시:** `Output` 함수 하나를 만들고 파라미터가 1이면 '프린터 출력', 2이면 '화면 출력'을 하게 만든 경우.

## 3. 시간적 응집도 (Temporal Cohesion)

특정 시점에 반드시 함께 실행되어야 하는 활동들을 하나의 모듈에 모은 상태입니다.

* **예시:** 프로그램 시작 시 호출되는 `Initialize` 함수 안에 '변수 초기화', '파일 열기', '로그인 화면 띄우기'를 모아둔 경우.

## 4. 절차적 응집도 (Procedural Cohesion)

모듈 안의 요소들이 순차적으로 실행되어야 하지만, 각 요소가 다루는 데이터는 서로 다른 경우입니다.

* **예시:** '사용자 권한 확인' 후 '파일 읽기'를 수행하는 모듈. (동작 순서는 맞지만 기능적 연관성은 낮음)

## 5. 통신적(교환적) 응집도 (Communication Cohesion)

동일한 입력 데이터를 사용하여 서로 다른 기능을 수행하는 요소들을 모은 상태입니다.

* **예시:** '회원 정보' 데이터를 받아서 '회원 등급 계산'도 하고 '회원 주소 출력'도 하는 모듈.

## 6. 순차적 응집도 (Sequential Cohesion)

한 요소의 출력값이 다음 요소의 입력값으로 사용되는 경우입니다.

* **예시:** '데이터 읽기' → '데이터 암호화' → '암호화된 데이터 저장' 과정이 하나의 모듈에 있는 경우.

## 7. 기능적 응집도 (Functional Cohesion) - **가장 높음**

모듈 내부의 모든 요소가 **단 하나의 목표**를 위해 긴밀하게 협력하는 상태입니다.

* **예시:** '삼각형의 넓이를 계산하는 함수'. 이 함수 안의 모든 코드는 오직 넓이 계산만을 위해 존재합니다.

---

### 💡 응집도 요약 비교

| 등급 | 종류 | 특징 | 평가 |
| --- | --- | --- | --- |
| **낮음** | 우연적, 논리적 | 관계가 없거나 억지로 묶임 | **나쁨** |
| **중간** | 시간적, 절차적, 통신적 | 시점이나 순서, 데이터 공유 중심 | **보통** |
| **높음** | 순차적, **기능적** | 하나의 목적을 위해 존재 | **매우 좋음** |

> **기억하세요!** 응집도가 높을수록 해당 모듈은 이해하기 쉽고, 재사용하기 좋으며, 수정 시 다른 곳에 영향을 줄 확률이 낮아집니다.
