
# FDD(Feature Driven Development)
>  **FDD(Feature Driven Development, 기능 중심 개발)**는 1997년 제프 드 루카(Jeff De Luca)가 제안한 방법론으로,
> 사용자에게 실질적인 가치를 제공하는 **'기능(Feature)'**을 중심으로 개발을 진행하는 애자일(Agile) 방식입니다.
> 대규모 프로젝트에서도 팀 간 정렬과 품질 확보가 용이하도록 설계된 것이 특징입니다.

---

## 1. FDD의 핵심 5단계 프로세스

FDD는 크게 '모델 중심의 초기 설계'와 '기능 중심의 반복 개발'로 나뉩니다. 전체 노력의 약 75%가 후반부의 반복 단계(4, 5단계)에 집중됩니다.

1. **전체 모델 개발 (Develop an Overall Model):** 도메인 전문가와 개발자가 협력하여 시스템의 전체적인 윤곽(도메인 모델)을 그립니다. 클래스 다이어그램 등을 통해 시스템의 구조를 시각화합니다.
2. **기능 목록 작성 (Build a Feature List):** 전체 모델을 바탕으로 사용자에게 가치를 주는 작은 기능들을 목록화합니다. 각 기능은 보통 **"2~10일 이내"**에 완료될 수 있을 정도로 작게 쪼개는 것이 원칙입니다.
3. **기능별 계획 (Plan by Feature):** 기능 리스트의 우선순위를 정하고, 각 기능을 담당할 개발자(Class Owner)를 할당하여 개발 일정을 수립합니다.
4. **기능별 설계 (Design by Feature):** 실제 코딩에 들어가기 전, 담당 프로그래머들이 모여 해당 기능에 대한 세부 설계를 진행하고 코드 리뷰(Design Inspection)를 거칩니다.
5. **기능별 구축 (Build by Feature):** 설계를 바탕으로 실제 코드를 구현하고 테스트합니다. 완료된 기능은 전체 시스템에 통합됩니다.

---

## 2. FDD의 주요 특징

* **사용자 가치 중심:** 사용자가 이해할 수 있는 언어로 표현된 '기능'을 최소 단위로 삼습니다. (예: "결제 수단을 선택할 수 있다")
* **도메인 모델링 기반:** 단순한 코딩보다 시스템의 비즈니스 로직(도메인)을 정확히 이해하고 모델링하는 것을 강조합니다.
* **클래스 소유권 (Class Ownership):** 각 코드 클래스마다 담당자(Owner)가 지정되어 있어 책임 소재가 명확하고 코드의 일관성을 유지하기 쉽습니다.
* **진척도 시각화:** 각 기능별로 6단계의 마일스톤(도메인 파악, 설계, 설계 검사, 코드 작성, 코드 검사, 빌드 통합)을 두어 프로젝트 진행률을 매우 정밀하게 추적합니다.

---

## 3. 다른 방법론과의 비교

| 구분 | FDD (기능 중심 개발) | Scrum (스크럼) |
| --- | --- | --- |
| **주요 단위** | 기능 (Feature) | 사용자 스토리 (User Story) |
| **반복 주기** | 기능당 2~10일 | 스프린트 (1~4주) |
| **팀 구조** | 역할 중심 (수석 프로그래머, 클래스 소유자 등) | 자율적, 다기능 팀 (Cross-functional) |
| **설계 방식** | 초기에 엄격한 도메인 모델링 수행 | 개발 과정에서 점진적으로 아키텍처 형성 |
| **적합성** | 대규모, 복잡한 비즈니스 시스템 | 변화가 잦고 빠른 배포가 필요한 소규모 팀 |

---

> **요약하자면:** FDD는 **"모델링은 정교하게, 구현은 기능 단위로 빠르게"** 수행하는 방식입니다. 체계적인 문서화와 명확한 역할 분담이 필요하면서도 애자일의 유연성을 잃고 싶지 않은 중대형 프로젝트에 매우 효과적입니다.
