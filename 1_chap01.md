# Section 001. 소프트웨어 생명 주기 (A 등급)

## 📌 소프트웨어 공학
- 소프트웨어의 위기를 극복하기 위한 방안으로 연구된 학문  
- 개발, 운용, 유지보수에 대한 체계적 접근  
- 품질과 생산성 향상 목적  
- 경제적 비용으로 신뢰성 높은 SW 개발을 위한 공학적 원리 적용  

## 📌 소프트웨어 공학의 기본 원칙
- 현대적 프로그래밍 기술 지속 적용  
- 품질 유지 위해 지속적 검증  
- 개발 관련 사항 및 결과 명확한 기록 유지  

---

## 📌 폭포수 모형(Waterfall Model)
- 이전 단계로 돌아갈 수 없다는 전제  
- 각 단계를 확실히 매듭짓고 승인 후 다음 단계 진행  
- 보헴(Boehm)의 고전적 생명주기 모형  
- 가장 오래되고 폭넓게 사용  
- 요구사항 반영이 어려움

## 📌 프로토타입 모형(Prototype Model)
- 요구사항 정확 파악 위해 시제품(Prototype) 제작  
- 의뢰자·개발자 모두의 참조 모델  
- 시스템 일부 또는 모형을 먼저 제작  
- 새로운 요구사항 발생 시 프로토타입 반복 제작  
- 단기간 제작으로 비효율적 언어/알고리즘 사용 가능

## 📌 나선형 모형(Spiral Model)
- 보헴(Boehm) 제안  
- 폭포수 + 프로토타입 + 위험 분석  
- 반복적 개발로 점진적 완성  
- 과정: 계획 → 위험 분석 → 개발·검증 → 고객 평가  
- 핵심 기술 문제나 요구사항 이해 어려울 때 적합

## 📌 애자일 모형(Agile Model)
- 요구사항 변화에 유연 대응  
- 반복적 개발 주기 기반  

### 애자일 기반 개발 모형
스크럼, XP, 칸반, 린, 크리스탈, ASD, FDD, DSDM, DAD 등

### 애자일 4대 가치
- 프로세스·도구보다 개인·상호작용  
- 문서보다 실행되는 SW  
- 계약 협상보다 고객 협업  
- 계획 준수보다 변화 대응

# Section 002. 스크럼(Scrum) 기법 (C 등급)

## 📌 스크럼 팀 구성
### ● 제품 책임자(PO)
- 요구사항 작성 및 우선순위 지정  
- 이해관계자 의견 종합  

### ● 스크럼 마스터(SM)
- 스크럼 수행을 위한 가이드 역할  

### ● 개발팀(DT)
- 개발자, 디자이너, 테스터 등 전체 팀원  

---

## 📌 스크럼 개발 프로세스
- **제품 백로그**: 모든 요구사항 목록  
- **스프린트 계획 회의**: 이번 스프린트 작업 선정  
- **스프린트**: 2~4주 개발 주기  
- **일일 스크럼 회의**: 15분 진행 상황 점검  
- **스프린트 검토 회의**: 결과물 테스트  
- **스프린트 회고**: 개선점 확인

## 📌 XP 핵심 가치

- 용기

- 피드백

- **소규모 릴리즈**  

## 📌 XP 실천 방법

- Collective Ownership

# Section 007. 요구사항 분석 (A 등급)

## 📌 요구사항 분석 개요
- 사용자 요구 이해 및 문서화  
- 요구 타당성 조사  
- 비용·일정 제약 설정  
- 모호성 제거  
- 상충 요구 조정  
- 사용 도구: UML, DFD, 자료사전, ERD 등  

---

## 📌 자료 흐름도(DFD) 구성 요소

| 기호 | 의미 |
|------|------|
| 프로세스 | 처리 |
| 자료 흐름 | 데이터 이동 |
| 자료 저장소 | 데이터 저장 |
| 단말 | 외부 개체 |

### 자료 흐름도 작성 지침
- 처리 후 변환되면 새로운 이름 부여  
- 출력 위해 입력 필요  
- 상위·하위 자료 흐름 일치  
- 입력 있다고 출력 반드시 필요한 것은 아님  

---

## 📌 자료 사전 표기 기호

| 기호 | 의미 |
|------|------|
| = | 자료 정의 |
| + | 연결(and) |
| () | 생략 가능 |
| [ ] | 선택(or) |
| {} | 반복 |
| ** ** | 주석 |

# Section 009. UML (A 등급)

- 객체지향 모델링 표준 언어  
- 구성요소: 사물, 관계, 다이어그램

- 구조 사물  
- 행동 사물  
- 그룹 사물  
- 주해 사물  

---

## 📌 관계
### ● 의존(Dependency)
- 짧은 시간 동안 영향 주는 관계  
- 매개변수로 다른 클래스를 사용할 때

- 기능 수행을 지정하는 관계  

### ● 일반화(Generalization)
- 상위·하위 개념 관계
