# SW개발 방법론
> **시대별로 드러난 문제를 해결하기 위해 등장한 일종의 진화 과정**
> 즉, *문제가 생기고 → 그 문제를 해결하는 새로운 방법론이 등장하고 → 또 새로운 문제가 생기고…* 이런 식으로 발전하였음

---

# 📘 소프트웨어 개발 방법론의 역사적 발전 과정  
## — “문제가 새로운 방법론을 낳고, 그 방법론이 또 다른 문제를 낳는다”

---

# 1️⃣ 1950~1960년대: **코드 중심 개발(Ad-hoc Development)**  
### ✔ 특징
- 문서도 없고, 절차도 없고, 그냥 개발자가 바로 코딩  
- “장인 정신” 기반 개발  
- 개인 역량에 의존  

### ❌ 문제점
- 유지보수 불가능  
- 개발자 교체 시 프로젝트 붕괴  
- 규모가 커지면 통제 불가  

### ➡ 해결을 위해 등장한 것  
→ **구조적 개발 방법론(Structured Programming)**  
(절차, 흐름 제어, 모듈화 개념 등장)

---

# 2️⃣ 1970년대: **폭포수 모형(Waterfall Model)**  
### ✔ 등장 배경
- 소프트웨어 위기(Software Crisis) 해결 필요  
- 개발을 “공학적 절차”로 관리하려는 시도  
- 문서화, 단계적 진행, 품질 관리 강조  

### ✔ 장점
- 체계적, 문서 중심  
- 대규모 프로젝트 관리 가능  

### ❌ 문제점
- 요구사항 변경이 거의 불가능  
- 고객은 결과물을 너무 늦게 봄  
- 실제 개발은 항상 변경이 발생하는데 모델은 변경을 허용하지 않음  

### ➡ 해결을 위해 등장한 것  
→ **프로토타입 모델, 점진적 개발 모델**

---

# 3️⃣ 1980년대: **프로토타입 모델 & 점진적 개발**  
### ✔ 등장 배경
- 고객이 요구사항을 명확히 설명하기 어려움  
- 폭포수 모델의 “변경 불가” 문제 해결 필요  

### ✔ 장점
- 시제품(Prototype)으로 요구사항 명확화  
- 고객과 빠른 피드백 가능  

### ❌ 문제점
- 프로토타입이 실제 시스템처럼 오해됨  
- 구조적 설계가 약해 품질 저하 가능  
- 반복이 많아 일정 관리 어려움  

### ➡ 해결을 위해 등장한 것  
→ **나선형 모델(Spiral Model)**  
→ **객체지향 개발 방법론(OOP)**

---

# 4️⃣ 1990년대: **나선형 모델(Spiral Model) & 객체지향(OOP)**  
### ✔ 등장 배경
- 대규모 프로젝트에서 위험 관리 필요  
- 반복적 개발 + 위험 분석 결합  
- 객체지향 패러다임 확산  

### ✔ 장점
- 위험 분석 기반  
- 반복적 개발로 요구사항 변경 수용  
- 객체지향으로 재사용성·유지보수성 향상  

### ❌ 문제점
- 복잡하고 비용이 많이 듦  
- 숙련된 인력 필요  
- 문서 중심 문화는 여전히 무거움  

### ➡ 해결을 위해 등장한 것  
→ **경량 개발 방법론(Agile)**  
→ **애자일 선언(2001)**

---

# 5️⃣ 2000년대: **애자일(Agile) 개발 방법론**  
### ✔ 등장 배경
- 인터넷 시대 → 요구사항 변화 속도 폭발  
- 고객 참여 증가  
- 기존 방법론은 너무 무겁고 느림  

### ✔ 핵심 가치
- 문서보다 **작동하는 소프트웨어**  
- 절차보다 **사람과 상호작용**  
- 계약보다 **고객 협업**  
- 계획보다 **변화 대응**  

### 대표 기법
- 스크럼(Scrum)  
- XP(extreme Programming)  
- 칸반(Kanban)  
- 린(Lean)  

### ❌ 문제점
- 문서 부족으로 인수인계 어려움  
- 팀 역량에 따라 품질 편차 큼  
- 대규모 프로젝트 적용 어려움  

### ➡ 해결을 위해 등장한 것  
→ **Scaled Agile Framework(SAFe)**  
→ **DevOps**

---

# 6️⃣ 2010년대~현재: **DevOps & CI/CD 시대**  
### ✔ 등장 배경
- 애자일은 개발만 빠르고 배포는 느림  
- 클라우드·자동화 기술 발전  
- 개발(Dev)과 운영(Ops)의 분리 문제  

### ✔ DevOps의 핵심
- 개발 + 운영 통합  
- 자동화(CI/CD)  
- 지속적 배포, 지속적 통합  
- 협업 문화 강조  

### ❌ 문제점
- 조직 문화 변화 필요  
- 자동화 도구 학습 비용  
- 모든 조직에 맞는 만능 해법은 아님  

### ➡ 이후 발전 방향  
→ **MLOps, DataOps, GitOps 등으로 확장**  
→ **AI 기반 개발(코드 자동 생성, 테스트 자동화)**

# 📌 전체 흐름을 한눈에 보는 요약 표

| 시대 | 방법론 | 해결하려던 문제 | 새로 발생한 문제 | 다음 방법론 |
|------|---------|------------------|-------------------|--------------|
| 1950s | Ad-hoc | 무질서한 개발 | 유지보수 불가 | 구조적 개발 |
| 1970s | 폭포수 | 체계적 관리 | 변경 불가 | 프로토타입 |
| 1980s | 프로토타입 | 요구 불명확 | 품질 저하 | 나선형 |
| 1990s | 나선형/OOP | 위험 관리 | 복잡·비용 증가 | 애자일 |
| 2000s | 애자일 | 변화 대응 | 문서 부족 | DevOps |
| 2010s~ | DevOps | 배포 자동화 | 조직 변화 필요 | MLOps 등 |

# ✨ 마무리  
소프트웨어 개발 방법론의 역사는 결국 **“변화하는 환경에 적응하기 위한 진화의 역사”**야.  
각 시대의 문제를 해결하려고 새로운 방법론이 등장했고, 그 방법론이 또 다른 문제를 낳으면서 다음 단계로 발전해 왔지.

원하면 이 내용을 **시험 대비 요약본**, **마인드맵**, **타임라인 그림**, **표 기반 암기 노트**로도 만들어줄 수 있어.
